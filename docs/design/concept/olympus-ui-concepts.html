Olympus — six UI directions

Six substantially different concepts for the Olympus control plane, each with a schematic layout sketch, a full breakdown, and honest trade-offs. A recommendation, MVP screen list, and component list follow at the end.

1 Mission Control 2 Workbench 3 Console 4 Dispatch Board 5 Ledger 6 Atlas Recommendation + MVP
01

Mission Control

SHIPPED BASELINE
RAIL · SESSION LIST · TRANSCRIPT · INSPECTOR

Design thesis. This is the shipped app — the design system's groundtruth — included as the baseline the other five directions depart from. The fleet is the product. One persistent cockpit where every running agent is visible at all times; everything else (projects, repos, artifacts) is context hanging off a session. Optimized for the supervising operator who has 6+ agents live and needs to know, at a glance, which one needs attention.

Visual personality. Dark instrument panel — the shipped Olympus "Instrument" language. Monochrome, silver accent, hairline borders, mono micro-labels, pastel status dots. Calm density; nothing animates unless it is alive.

Primary layout. Fixed shell: 34px topbar (org pill, global search, runtime status), 220px left rail, master session list (~280px), transcript center column (760px measure), 279px right inspector, collapsible bottom terminal panel.

Navigation. Flat left rail (Sessions, Board, Nodes, Workflows, Vaults, Usage, Settings) + ⌘K palette for everything. Sessions are the home screen; you rarely leave it.

Key screens. Sessions (master-detail), Nodes fleet grid, Board, Workflow runs, Usage, Search.

Sessions. Rows in the master list: status dot, title, source channel dot, repo, relative time, token count. Selected session fills the center as a transcript with tool-call cards and diffs.

Agents. An agent is the session's engine, shown in the inspector: model, node, slot, posture (managed / observed), controls (pause, stop, fork).

Structure. Org pill in topbar; project + repo are metadata chips on each session and filters on the list — hierarchy is flattened into filters, not a tree.

Live work. Streaming transcript with a blinking caret, pulsing live dots in the list, bottom panel tails raw terminal/logs, inspector shows tokens/sec and current tool call.

Artifacts & history. An Artifacts tab in the inspector per session; global Search spans all transcripts. History = the session list filtered to done/failed.

Trust. Observed sessions are read-only with an explicit fork banner. Destructive actions (stop, delete, force-push) get typed-confirmation dialogs; permission scopes shown as mono badges on the session.

STRENGTHS
Best at-a-glance fleet awareness · matches shipped app (lowest risk) · everything ≤2 clicks from a running session
WEAKNESSES
Project/repo structure is weakly expressed · dense shell intimidates new users · artifacts feel secondary

Best for: the power operator running many concurrent agents daily.

02

Workbench

IDE MODEL
REPO TREE · DIFF EDITOR · AGENT PANEL · TERMINAL

Design thesis. The code is the product; the agent is a collaborator inside your editor. Instead of watching a transcript, you live in the repo — file tree, diff-first editor, terminal — and the agent's activity is annotated onto the code itself.

Visual personality. Same Instrument tokens, but editor-dense: tabbed documents, gutter markers, syntax-height rhythm. Feels like a serious IDE, not a dashboard.

Primary layout. IDE trichotomy: repo tree left, tabbed diff/editor center, agent panel right (transcript + controls collapsed into a column), terminal strip bottom.

Navigation. Workspace switcher (project+repo pairs) in the topbar; ⌘P file jump, ⌘K commands. Sessions appear as "agent tabs" alongside file tabs.

Key screens. Workspace (the IDE), Changes review (stacked diffs per session), Runs (terminal history), a minimal home listing workspaces.

Sessions & agents. A session = an agent tab pinned to a workspace. The agent panel shows its current tool call, plan checklist, and a compact transcript; file-tree rows glow where the agent is writing.

Structure. Fully explicit — the repo tree IS the structure. Projects are workspace groups; vaults appear as a locked node in the tree.

Live work & observability. Live diffs stream into the editor; the terminal tails stdout; gutter badges mark agent-touched lines with provenance on hover.

Artifacts & history. Changes review is the artifact surface: every session yields a reviewable changeset; older sessions reopen as read-only workspaces.

Trust. Write access is per-workspace and visible in the tab (a lock glyph); agent edits land as staged changes you approve per-file, git-style. Dangerous shell commands require inline approval in the terminal.

STRENGTHS
Deepest code inspection · per-file approval is a natural trust model · familiar to every developer
WEAKNESSES
One workspace at a time — poor fleet awareness · heavy to build · duplicates the user's real IDE

Best for: the hands-on reviewer who supervises one or two agents closely and reads every diff.

03

Console

TERMINAL-NATIVE
SPLIT PANES · PROMPT LINE · STATUS BAR

Design thesis. Text is truth. Everything — launching agents, tailing sessions, querying artifacts — happens through a command line and streaming panes, tmux-style. GUI chrome is reduced to a status bar and a pane manager; the keyboard is the whole interface.

Visual personality. Purest expression of the Instrument language: near-black, all Plex Mono, color used only for stdout semantics. Zero decorative chrome.

Primary layout. Full-bleed splittable panes (each pane = a session tail, log query, or shell), one persistent prompt line, a bottom status bar listing pane "windows".

Navigation. Commands (`ol new`, `ol attach 42`, `ol ps`, `ol logs --follow`) plus keybindings to split/focus panes. No sidebar, no tabs — a fuzzy-find overlay lists sessions.

Key screens. There is one screen; "views" are pane arrangements saved as layouts (fleet ps, session tail, diff pager, usage table).

Sessions & agents. `ol ps` renders a live table (id, status, node, model, tok/s). Attaching turns a pane into that session's stream; the agent is addressed by talking on its prompt line.

Structure. Path-like addressing: `org/project/repo#session`. `ol tree` prints the hierarchy; completion makes it navigable.

Live work & observability. The native medium — everything is already a stream. Structured events render as colored, greppable lines; `ol trace 42` pretty-prints a tool-call tree inline.

Artifacts & history. `ol artifacts`, `ol show patch-7` piped into a pager pane; history is shell history plus full-text `ol grep` across transcripts.

Trust. Dangerous commands echo an explicit consequence line and require re-typing the target id; permissions are flags visible in `ol ps`. Everything is logged and replayable by design.

STRENGTHS
Fastest for experts · scriptable and composable · trivially local-first · cheap to build on existing CLI
WEAKNESSES
Steep learning curve · poor for diffs, boards, rich artifacts · discoverability near zero

Best for: terminal loyalists who already live in tmux and want Olympus as an amplifier, not a GUI.

04

Dispatch Board

WORK QUEUE
TODO · READY · RUNNING · BLOCKED · DETAIL DRAWER

Design thesis. You are not a babysitter, you are a dispatcher. Work is the unit, not the session: you write tasks, the system assigns them to agents, and the board is the single source of truth for what's queued, running, blocked, and done. Sessions become implementation detail behind cards.

Visual personality. Instrument tokens applied to a calm kanban: hairline columns, status-tinted card borders, mono badges for assignee-agent and cost. More whitespace than the cockpit.

Primary layout. Full-width board (todo → ready → running → blocked → done), right detail drawer for the selected card, thin topbar with project filter and a prominent "New task".

Navigation. Board is home. Secondary: Agents roster, Artifacts library, Activity feed. Cards deep-link into a session transcript overlay when you need to look under the hood.

Key screens. Board, Card detail (task, assigned agent, live progress, produced artifacts), Agents roster, Activity feed.

Sessions. Hidden by default — each running card shows a live one-line status ("editing auth.ts · 12k tok") and expands to a transcript overlay on demand.

Agents. First-class workers with a roster page: capacity, specialties (coder / reviewer / researcher), current assignment, success rate. Cards show the assignee as a mono chip.

Structure. Projects are board swimlanes or filters; repo is a field on the card. Structure serves routing, not browsing.

Live work & observability. Cards move columns on live events; running cards carry progress lines and a pulsing dot; the Activity feed is a chronological event stream across all cards.

Artifacts & history. Done cards accumulate their outputs (patch, report, build) as attachments; the Artifacts library indexes them across cards. History is the done column plus feed.

Trust. Blocked column is the trust mechanism: any action needing approval parks the card in blocked with a specific question; you approve/deny from the drawer. Per-project permission presets decide what auto-runs.

STRENGTHS
Scales to many agents without attention overload · blocked-as-inbox is a clean approval model · legible to non-operators
WEAKNESSES
Indirection frustrates users who want to steer mid-run · conversational work fits cards poorly · observability is second-hand

Best for: the delegator running high volumes of well-specified tasks who checks in twice a day.

05

Ledger

ARTIFACT-FIRST ARCHIVE
SEARCH BAR · TIME SPINE · ARTIFACT CARDS

Design thesis. What matters is what was produced, provably. Olympus as a durable, append-only ledger of outcomes: every patch, report, build, and decision on one searchable timeline with full provenance. Live supervision exists, but the archive is the center of gravity — the system of record for AI work.

Visual personality. Quietest of the six: editorial spacing, a single vertical time spine, document-like artifact cards. The mono/caps kicker system does the wayfinding.

Primary layout. A centered chronological spine (today at top), grouped by day; dominant search/filter bar; artifact detail opens as a full-height right panel with a provenance strip.

Navigation. Search-first: type-to-filter across artifacts, sessions, repos. Facet pills (type, project, agent, date). "Now" is just the newest ledger entries, marked live.

Key screens. Ledger (home), Artifact detail + provenance, Session replay (a read-only transcript reconstructed from the ledger), Collections (pinned sets).

Sessions. A session is a span on the timeline — a bracket grouping its events and outputs. Replay scrubs through it; fork resumes it.

Agents. Bylines. Every entry is signed "agent · model · node"; an agent page is the ledger filtered to its byline with an accuracy/cost summary.

Structure. Facets, not folders — project and repo are provenance fields on entries. Collections give curated project views.

Live work & observability. The top of the spine is live: running sessions append entries in real time. Depth-first inspection happens in replay: every tool call and file touch is an immutable event you can step through.

Artifacts & history. The whole product. Diff viewers, report renderers, build manifests inline; every artifact answers "what changed, who changed it, where it lives" in one provenance strip.

Trust. Strongest story: append-only history, signed provenance, visible sandbox boundary per entry. Approvals are themselves ledger entries — who approved what, when, is permanent.

STRENGTHS
Unmatched auditability and recall · provenance builds durable trust · resuming old work is natural
WEAKNESSES
Weak at real-time steering · feels retrospective when you want a cockpit · empty on day one

Best for: teams and compliance-minded users who treat agent output as records that must be reviewable months later.

06

Atlas

SPATIAL PROJECT GRAPH
PANNABLE CANVAS · ENTITY NODES · LIVE EDGES · MINIMAP

Design thesis. The system is a graph — show it as one. Projects, repos, sessions, agents, vaults, and artifacts are nodes on an infinite canvas; edges are real relationships (runs-on, writes-to, produced, reads-from). Understanding "what is connected to what" becomes perception instead of navigation.

Visual personality. Instrument tokens on a deep-black canvas: hairline edges, softly-squared node cards, status expressed as border tint and pulse. Feels like a network operations map.

Primary layout. Full-viewport pan/zoom canvas, floating filter pills top-left, minimap bottom-right, and a right inspector that opens when a node is selected (and hosts the transcript for session nodes).

Navigation. Spatial: zoom out for the org, zoom into a project cluster, double-click a session node to expand it in place. ⌘K jumps the camera to any entity. Saved viewpoints act like bookmarks.

Key screens. One canvas with lenses: Topology (default), Activity (edges animate with event flow), Provenance (select an artifact, its ancestry lights up), plus the node inspector.

Sessions & agents. Sessions are nodes tethered to their repo and node-host; running ones pulse and stream a one-line status. Agents are the engine badge on the session node; the fleet is visible as host clusters.

Structure. Literal — containment and attachment are drawn. Orphaned repos, shared vaults, and cross-project dependencies are visible at a glance, which no list view shows.

Live work & observability. The Activity lens animates events along edges (tool call → repo write → artifact born). Failures flash red at the exact node; tracing a cascade is following lit edges.

Artifacts & history. Artifacts are terminal nodes hanging off sessions; the Provenance lens is the review tool. Time-travel scrubber replays the graph state at any past moment.

Trust. Sandbox boundaries are drawn as literal enclosures around nodes; a session touching outside its boundary is visually crossing a line. Destructive actions preview their blast radius on the graph before confirming.

STRENGTHS
Makes structure and blast radius perceivable · unique identity · brilliant for debugging cross-entity cascades
WEAKNESSES
Spatial layouts degrade at scale · slow for routine tasks vs a list · hardest to build well · accessibility challenges

Best for: architects of complex multi-repo, multi-agent systems who need to reason about relationships and containment.

Recommendation

Ship Mission Control — with the Board and the Ledger's provenance strip folded in

Why. Mission Control is already the shipped groundtruth — the design system was extracted from it, so v1 carries zero visual or interaction risk. It directly serves the core loop (launch → watch → steer → inspect), gives the best multi-agent awareness per pixel, and every other concept degrades gracefully into it: the Board is already a view in the shell, the Console lives on as the bottom terminal panel + ⌘K, and the Ledger's provenance strip can be adopted as a per-artifact detail without a new navigation model. Workbench and Atlas are the two that demand a different skeleton — defer them.

What to steal for v1: the Dispatch Board's blocked-as-approval-inbox (concept 4) as the trust surface, and the Ledger's provenance strip (concept 5) on every artifact: produced by · session · model · node · repo@sha · stored at.

MVP SCREENS
  1. Sessions — master list + transcript + composer + right inspector (home)
  2. New session dialog — repo, agent/model, permission preset
  3. Board — 5-column kanban wired to live card events; blocked = approval inbox
  4. Session artifacts tab — outputs with provenance strip + diff viewer
  5. Nodes — fleet grid: heartbeat, slots, posture
  6. Search — full-text across sessions, grouped by source
  7. Settings — theme, density, runtime token, model routing
  8. ⌘K palette + bottom terminal panel — cross-cutting, not separate screens
