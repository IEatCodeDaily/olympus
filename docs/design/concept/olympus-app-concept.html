PR#142 · open
The auth gate needs the loopback-origin check before it hits the token compare — right now a remote origin with a stolen token still passes. Can you reorder and add an origin allowlist?
CLAUDE-CODE

Good catch — the Origin check currently runs after the bearer compare, so a valid token from a non-loopback origin is accepted. I'll reorder so the gate fails closed on origin first, then compares the token.

Plan for auth.rs:

  1. Parse Origin / Host and reject anything not in the loopback allowlist (127.0.0.1, ::1, configured remote bind).
  2. Only then constant-time compare the per-install token.
  3. Return 403 (not 401) on origin failure so the client doesn't retry with creds.
