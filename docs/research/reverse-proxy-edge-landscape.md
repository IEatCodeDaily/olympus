# Reverse-proxy / local HTTP-edge landscape for Olympus

**Research date:** 2026-07-12. **Target:** an externally owned, locally supervised edge on one Ubuntu host, later multiple nodes; Olympus creates/deletes ephemeral routes, serves artifact files, and delegates every protected request to its own authorization API.

## Evaluation method and security caveats

1. **Hard filters were interpreted literally.** A graceful or “zero-downtime” reload is still a reload. Rewriting a watched YAML/TOML file is still a file rewrite. An internal UI API is not automatically a stable public automation contract.
2. **CVE window:** NVD publication dates from 2023-07-12 through 2026-07-12, cross-checked against project advisory pages. Search starting points: [NVD HAProxy](https://nvd.nist.gov/vuln/search/results?query=HAProxy&search_type=all), [Envoy](https://nvd.nist.gov/vuln/search/results?query=Envoy%20Proxy&search_type=all), [Caddy](https://nvd.nist.gov/vuln/search/results?query=Caddy%20web%20server&search_type=all), [Traefik](https://nvd.nist.gov/vuln/search/results?query=Traefik&search_type=all), [NPM](https://nvd.nist.gov/vuln/search/results?query=Nginx%20Proxy%20Manager&search_type=all), and the [official nginx](https://nginx.org/en/security_advisories.html) and [Apache httpd](https://httpd.apache.org/security/vulnerabilities_24.html) lists.
3. **Count inclusion:** command injection/arbitrary write with demonstrated or direct RCE potential; buffer overflow/underflow/overread, UAF, double-free, OOB read/write, and stack-overflow-class defects. Pure DoS/resource exhaustion, request smuggling, auth bypass, XSS, SSRF, and dependency-only CVEs were not counted. Thus **0 does not mean secure**. For example Pingora has three 2026 request-smuggling/cache CVEs, oauth2-proxy has recent auth bypasses, and Caddy has CVE-2026-27589 involving its local Admin API, but none belong in this deliberately narrow count.
4. **Exposure is configuration-dependent.** Several nginx defects require HTTP/3 or optional modules; many Envoy defects require a specific filter; several Apache defects require optional proxy modules or a malicious upstream. Counts measure security-response burden and unsafe-code surface, not expected exploit frequency in Olympus’s exact config.
5. **Dependencies matter operationally even when excluded from counts.** Go/Rust projects can still inherit crypto/parser CVEs; Pomerium embeds Envoy; OpenResty/APISIX/Kong inherit nginx/LuaJIT risk. Pin and scan the full shipped artifact/SBOM.
6. **Response-time evidence is uneven.** Apache’s page publishes report/fix/release dates (examples in 2026 range from roughly one day to three months). Envoy typically fixes several maintained branches in coordinated releases and has a Linux Foundation security team/Google patch-reward process. nginx routes reports through F5 SIRT. For smaller Rust/Go projects, “0 CVEs” often means a shorter exposure history and smaller researcher audience, not proven faster response.

## Per-candidate notes

### 1. Caddy — the only mature candidate that natively closes both discriminating gaps

Caddy’s Admin API is not a wrapper around config-file generation. Its native JSON is the runtime object model; `POST`/`PUT`/`PATCH`/`DELETE` mutate subtrees, individual requests are ACID, updates are provisioned before commit, failure rolls back, and traffic continues. Olympus should assign stable `@id` values to each session route and make one atomic subtree update per session. Caddy persists the latest successful configuration automatically; `--resume` restores it after restart.

It also has the exact adjacent primitives: `file_server`, `forward_auth`, WebSocket-aware reverse proxying, streaming/full duplex, and automatic HTTPS. Its on-demand TLS `ask` endpoint is particularly aligned: Olympus can authorize certificate issuance for a hostname before Caddy asks an ACME CA, preventing an attacker from consuming rate limits for arbitrary SNI.

Operationally, bind the Admin API to a Unix socket or loopback, do not expose it through public routes, and make Olympus the only writer. The API documentation labels the multi-request concurrency approach experimental, so perform one atomic route-array/subtree replacement or serialize writes; do not implement read-modify-write from multiple controllers.

**Security ownership:** Caddy maintainers own the server and release binary; Go owns the runtime/stdlib. The reviewed period had zero project-core RCE/memory-corruption-class CVEs, though CVE-2026-27589 is a reminder that the local Admin API is privileged and browser-origin hardening matters. Go removes classic UAF/buffer-overflow risk from ordinary Caddy code, but plugins and unsafe/crypto dependencies still require scanning.

### 2. Envoy — strongest control-plane protocol, weakest match on the two operator discriminators

xDS is the most mature multi-node dynamic configuration protocol in this set. It provides ACK/NACK, versioned resources, state-of-the-world and delta transports, removal semantics, warming, and a natural path from one Envoy to a fleet. `ext_authz` is richer than most forward-auth implementations and supports HTTP and gRPC, per-route settings, metadata, request-body limits, fail-open/closed choices, and header propagation.

The cost is substantial. Olympus must implement or embed an xDS management server and reason about LDS/RDS/CDS/EDS consistency. Envoy does not serve a filesystem tree, so artifacts need a second server/process (or Olympus itself), violating the stated single external-edge preference. Envoy does no ACME; SDS is certificate delivery, not issuance. Packaging is also less polished on bare Ubuntu now that the official apt repository is unmaintained, though static binaries and containers are available.

**Security ownership:** response ownership is excellent—CNCF/Linux Foundation plus major vendors, coordinated supported-branch releases, clear advisories—but the proxy is a huge C++ codebase. The narrow review found 18 memory-safety-class issues, usually configuration-specific and crash-oriented, with CVE-2026-48706 explicitly noting heap-write overflow and potential RCE. If memory safety is heavily weighted, “a strong team patches C++ quickly” is still worse prevention than a mature Go/Rust edge.

### 3. HAProxy — runtime API progress does not fully overturn the earlier finding

HAProxy’s current Runtime API is much more capable than old comparisons imply: dynamic backends and servers can be added/deleted, ACL/map entries can change transactionally, and certificates can be updated in memory. A design that predeclares a generic frontend and routes `Host` through a runtime map to predeclared/dynamic backends may satisfy Olympus without a reload. That is a valid constrained architecture.

It is not equivalent to arbitrary route CRUD. Structural rule/listener changes still go through the separate Data Plane API, whose own invocation example requires write access to `haproxy.cfg` and a reload command. Static artifact service and generic forward-auth are also not first-class. Lua/SPOE can fill both gaps, but then Olympus owns more proxy extension code and its security behavior.

HAProxy’s C security record in this narrow period is notably better than nginx/Apache/Envoy: one medium heap overflow in an uncommon `regsub` configuration. HAProxy Technologies and the community maintain LTS lines actively. It remains an excellent conventional load balancer, just not the cleanest fit for this control-plane contract.

### 4. nginx family

* **nginx OSS:** The earlier decision is correct. It is an excellent static server and `auth_request` exactly models forward-auth. It still has no OSS route CRUD API. A graceful reload is reliable and normally zero-downtime, but sessions several times per hour would make config generation/reload part of Olympus’s correctness boundary. NGINX Plus narrows the gap for upstream membership and key-value routing, not arbitrary location/server structure.
* **OpenResty:** One can build a genuinely dynamic router in Lua backed by shared dictionaries and expose an Olympus-facing endpoint. That is architecture work, not a stable upstream contract. It undermines the stated reason to externalize the proxy: Olympus becomes owner of route-table concurrency, persistence, validation and part of request parsing/auth logic.
* **NGINX Unit:** Its live Control API plus static service made it a serious conceptual fit, and it refutes any blanket claim that “all nginx-family products require reload.” However, Unit itself says it is archived and unmaintained as of October 2025. That directly violates the requirement that an external upstream own future CVE response.
* **Nginx Proxy Manager:** management UI/API does not change nginx’s underlying control plane. It generates files and reloads nginx, while adding Node, a database, privileged certificate automation, and two recent RCE-class command-injection issues. It is a homelab convenience layer, not an architecture simplifier here.

### 5. Traefik — the earlier “dynamic yes, static no” judgment stands

Traefik’s supported provider model satisfies no-reload dynamic desired state when Olympus writes a supported KV store (or manages Docker/Kubernetes metadata). Traefik also still contains a direct REST provider that accepts `PUT /api/providers/{provider}`, but current documentation barely exposes it; a 2026 advisory shows why its `insecure=false` boundary matters. Treat that write endpoint as an unstable/internal integration unless Traefik Labs explicitly commits to it. For the ADR, the honest trade is therefore either an extra KV process or dependence on an under-documented handler—not the clean, stable Caddy Admin API contract.

The blocker is static service. Traefik intentionally routes to services; it does not map a URL path to local disk. A tiny static sidecar would fix this operationally, but then the deliverable is two supervised processes. ForwardAuth and ACME are excellent. Traefik is Go and has broad commercial/community ownership, but recent advisories—especially route/auth bypasses outside the narrow CVE count—mean the ForwardAuth integration must be tested adversarially across path normalization and forwarded-header behavior.

### 6. Microsoft YARP

YARP can satisfy every functional item if Olympus authors a small ASP.NET Core host: an in-memory `IProxyConfigProvider`, Static Files middleware, a custom authorization middleware that calls Olympus, Kestrel protocols, and an ACME library or external issuer. That host can expose exactly the desired route API.

But this changes the buy/build boundary. Microsoft owns YARP and .NET vulnerabilities; the Olympus team owns composition order, auth fail-open behavior, the route API, persistence, certificate integration, packaging, and edge-level tests. It is less independent than using Caddy and adds a .NET runtime to a Rust product. It is a good choice only if deep custom behavior is strategically valuable.

### 7. Rust entrants: Pingora, River, Sōzu and rpxy

* **Pingora:** Cloudflare operates it at extraordinary scale, and Rust is attractive, but open-source Pingora is a framework. Selecting it means building the edge—the opposite of externalizing security ownership. Framework CVE response does not cover Olympus’s routing/auth/static application logic.
* **River:** It was explicitly intended to turn Pingora into a memory-safe general reverse proxy and reached hot-reload plus basic static serving. The published roadmap still places ACME and a stable external dynamic API in the future; the README says v0.5.0 has no stability expectation and file-only configuration. No project activity since 2024 makes it non-viable now.
* **Sōzu:** This is the most credible ready Rust proxy here. Its Unix command socket and protobuf command library are a genuine first-class, no-reload runtime control interface, and Clever Cloud is shipping signed binaries actively. However it does not natively provide artifact file serving or generic forward-auth, and its AGPL-3.0 license needs review if Olympus productizes or modifies it. The command library’s LGPL helps an external controller, but functionality remains missing.
* **rpxy:** Strong modern protocol/ACME feature set in a small Rust binary, but route changes are a watched TOML rewrite, forward-auth/static are absent, and the maintainer explicitly describes it as WIP/best-effort and personally driven.

### 8. Legacy/specialized servers

* **lighttpd:** superb small static server, but file/reload configuration and no modern control API.
* **H2O:** technically advanced H2/H3/static/proxy server, but no runtime route API and upstream has announced no more releases.
* **Varnish:** VCL can be loaded without restarting the daemon, but that is generated configuration. It is primarily an HTTP cache and needs a TLS terminator in OSS; it neither serves Olympus artifact directories nor owns ACME.
* **Apache Traffic Server:** similarly optimized for proxy/cache/CDN workloads; remap configuration reload and plugin complexity make it a poor local app router.
* **sniproxy:** TLS passthrough cannot call HTTP auth or serve HTTP files; eliminate before scoring.

### 9. Identity-aware and management layers

* **Pomerium:** a high-quality identity-aware proxy, but Olympus explicitly wants its own API consulted per request. Pomerium would introduce an IdP/session/policy/databroker control plane and embedded Envoy while still lacking artifact files and simple local route CRUD in Core. It solves a different problem.
* **oauth2-proxy:** useful as the endpoint that nginx/Traefik/Caddy consults, not as this edge. It has no dynamic routing API or ACME. Its limited static mode does not rescue the architecture.
* **Zoraxy:** unusually broad all-in-one homelab features, but weak public API guarantees, concentrated maintenance, AGPL, and two RCE-capable management features in the window are poor matches for a security-sensitive control plane.

### 10. API gateways and dynamic routers

* **Skipper:** technically credible; route sources update at runtime and Zalando uses it. Innkeeper/etcd is another component, static/ACME are absent, and the ecosystem is much smaller than Traefik/Envoy/Caddy.
* **Apache APISIX:** the Admin API is genuinely stable route CRUD and changes are real-time. The forward-auth and ACME plugins are relevant. Mandatory etcd plus OpenResty makes it multiple processes and exposes the C nginx/LuaJIT dependency surface; no static file origin.
* **Kong:** similarly mature Admin API in database mode, but adds a database/control plane and still lacks native static service. Its feature/licensing boundary between OSS and Enterprise must be checked plugin-by-plugin; there has not been a HashiCorp-style relicensing of the Apache-2.0 OSS core in this review.

## Adversarial top three

### 1. Caddy — recommend for the ADR and prototype

**Why it wins:** It is the only mature, actively maintained external process in the survey that simultaneously offers a first-class live route API, native artifact files, native per-request forward-auth, excellent WebSocket/H2/H3/streaming, and automatic/on-demand TLS. Go materially reduces memory-corruption risk, packaging is one binary, and multi-node evolution can use one Caddy per node with Olympus broadcasting the same atomic desired state; distributed certificate storage is available through storage modules.

**Strongest argument against it:** The Admin API’s object model becomes a vendor-specific control-plane contract, and leadership/bus factor is less diversified than Envoy/nginx. Multi-node config distribution is **not** itself a consensus/control-plane product: Olympus must own fan-out, idempotency, convergence, rollback visibility, and certificate-storage design. Also, safe concurrent multi-request config edits remain an area where the docs explicitly warn about collisions; use serialized/atomic subtree updates.

**Required proof of concept:** 1,000 create/delete cycles while WebSockets and streaming responses remain open; crash/restart with `--resume`; forward-auth deny/allow/header propagation and fail-closed timeout tests; path traversal/symlink tests for artifact roots; on-demand-TLS `ask` authorization and rate-limit tests; two-node convergence with one node unavailable.

### 2. Envoy — choose only if multi-node control-plane rigor outranks static/memory safety

**Why it is second:** xDS is by far the strongest long-term fleet protocol, and `ext_authz` is the richest auth-delegation contract. If Olympus expects a serious multi-node data plane, delta xDS, ACK/NACK, versioning and resource warming reduce future reinvention.

**Strongest argument against it:** It directly fails the two previously identified discriminators: no filesystem server and no ACME. It therefore requires at least an artifact sidecar and certificate controller. More importantly for this ADR, it is a large memory-unsafe C++ edge with 18 reviewed memory-safety-class CVEs in three years, including one potential RCE. “Upstream responds quickly” is good incident response, not memory-safety prevention. xDS also creates substantially more Rust control-plane code than Caddy’s JSON API.

**Decision condition:** Select Envoy only if the team explicitly accepts (a) a second artifact server, (b) external certificate management, (c) the C++ security burden, and (d) implementing/testing a real xDS server now for fleet-scale benefits later.

### 3. Traefik — operationally simple alternative if one tiny artifact sidecar is acceptable

**Why it is third:** Its provider model applies dynamic desired state without reloads or config-file rewrites; Olympus can write a supported KV store, while the under-documented direct REST provider is available only if explicitly accepted as a compatibility risk. ForwardAuth, WebSockets/streaming, automatic ACME and a single Go data-plane binary are mature. It is much simpler than Envoy/APISIX/Kong and has broad upstream ownership.

**Strongest argument against it:** The earlier static-file objection is fully valid. Adding a file server is architecturally small but destroys the “one supervised edge process” property and creates route/auth boundary questions (does auth happen before the sidecar, can files be reached directly, who sets cache/range headers?). Traefik also has a recurring class of path-normalization/forwarded-header/auth-bypass advisories; because Olympus depends on ForwardAuth as a security boundary, this history deserves more weight than its narrow memory/RCE count of one.

**Decision condition:** Pick Traefik over Caddy only if provider-driven desired state is preferred to direct Admin API mutation and the team is comfortable supervising a loopback-only static server.

## Decision

**Use Caddy 2 as the default ADR choice.** The prior discriminating-axis findings are accurate: **Traefik is dynamically configurable but is not a static file server; nginx is a static file server with excellent `auth_request` but nginx OSS has no first-class arbitrary route CRUD API.** HAProxy’s newer Runtime API narrows—but does not erase—the dynamic gap. NGINX Unit once combined dynamic API and static service, but its 2025 archival makes it unacceptable under the explicit external-CVE-owner requirement.

Keep the Olympus-to-edge interface vendor-neutral internally: define a Rust `EdgeDriver` around desired `Route { id, host/path, upstream, auth_policy, artifact_root, tls_policy }` state, implement Caddy first, and preserve the option for a future xDS driver. Do not expose Caddy’s JSON shape throughout the domain model.
